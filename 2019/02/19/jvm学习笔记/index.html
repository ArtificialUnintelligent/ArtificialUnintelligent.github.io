简单地做下jvm的学习笔记

概念

Jvm

java虚拟机（java virtual machine）缩写，java使用jvm屏蔽与平台相关的信息，做到平台无关性。jvm执行字节码时把字节码解释成具体平台上的机器指令执行


JDK

java语言软件开发工具包（java development kit）（注：软件开发工具包software development kit）

jdk包含：javac、jar、Javadoc、jdb、java、appletviewer、javah、javap、jconsole


JRE

java运行环境（java runtime environment）运行java所需的环境集合，包括：java runtime environment（java平台，不包含jdk）和java plug-in（用来跑java小程序）

基本结构：

运行时结构：

从上图可以看出，程序计数器、java虚拟机栈、本地方法栈为线程私有，生命周期与线程同

  1. 程序计数器

    当前线程所执行的字节码行号指示器

    独立存储，cpu多线程机制，为了线程切换后的复位

    记录正在执行的java方法的字节码指令地址，如果执行的是native方法，计数器值为空

  2. java虚拟机栈

    描述java方法执行的内存模型，每个方法执行的时候创建栈帧存储局部变量表、操作数栈、动态链接、方法出口等信息，每个方法从被调执行至完成就对应一个栈帧在虚拟机栈中从入栈到出栈的过程

  3. 本地方法栈

    与虚拟机栈类似，为native方法服务

java堆（Java Heap）与方法区（Method Area）为共有

  1. java堆（GC堆）

    所有线程共享

    虚拟机启动时创建

    存放对象实例（几乎所有对象实例及数组）

    采用分代收集算法（java堆分为新生代和老年代，新生代又有Eden、From survivor、To Survivor）

    可以处于不连续的内存空间，可扩展（通过控制-Xms和-Xmx）

  2. 方法区

    所有线程共享

    虚拟机启动时创建

    存放已被虚拟机加载的类信息、常量、静态变量、JIT编译后的代码等

    不需要连续的内存空间，可扩展、可固定大小，可不实现垃圾收集

    运行时常量池：

    方法区的一部分，存放编译期生成的各种字面量和符号引用，详见java虚拟机：运行时常量池

直接内存

不是虚拟机运行时数据区的一部分，也不是jvm规范中定义的内存区域，这部分内存也被频繁使用



JVM内存模型（JMM）

java1.8之后移除了永久带，引入Metaspace（元空间）

传送门：Java8 对应的 JMM 划分

JMM中的并发编程

两个概念

  1. 线程间通信

    线程间可通过共享内存和消息传递两种方式来进行。

    共享内存：线程间通过读写内存中的公共状态来隐式通信，例如通过共享对象进行通信

    消息传递：线程之前没有公共状态，通过发送消息来进行显式通信，例如wait()和notify()

  2. 线程间同步

    程序用于控制不同线程之间操作发生相对顺序的机制

    共享内存模型：同步显式进行，通过指定某个方法或代码段需要在线程间互斥执行

    消息传递模型：消息的发送在接收之前，因此为隐式进行

Java内存模型

概念：jmm定义线程和主内存之间的抽象关系是线程之间的共享变量存储在主内存中，每个线程都有一个私有的本地内存，本地内存中存储该线程读写共享变量的副本（本地内存是一个抽象概念，不真实存在）

  3. static变量以及类本身相关信息

jvm桥接硬件架构

jmm和硬件内存架构是交叉关系，由此产生两个问题

  1. 共享对象的可见性：

    多线程操作同一共享对象时，在没有并发控制的情况下，一个线程的共享对象的更新可能导致其他线程看不见

    可以使用volatile，变量直接从主存读取，更新也直接写到主存。基于Memory Barrier实现

  2. 竞争现象：

    多个线程同时修改同一共享对象

    使用synchronized关键字，保证同一个时刻只能有一个线程进入代码竞争区。

  3. volatile和 synchronized区别

    • volatile本质是在告诉jvm当前变量在寄存器（工作内存）中的值是不确定的，需要从主存中读取； synchronized则是锁定当前变量，只有当前线程可以访问该变量，其他线程被阻塞住。

    • volatile仅能使用在变量级别；synchronized则可以使用在变量、方法、和类级别的

    • volatile仅能实现变量的修改可见性，不能保证原子性；而synchronized则可以保证变量的修改可见性和原子性

    • volatile不会造成线程的阻塞；synchronized可能会造成线程的阻塞。

    • volatile标记的变量不会被编译器优化；synchronized标记的变量可以被编译器优化

支撑jmm的基础原理

  1. 指令重排序：编译器在不改变单线程程序语义的前提下重新安排语句的执行顺序；如果不存在数据依赖性，处理器可以改变语句对应机器指令的执行顺序
  2. 内存屏障（memory barrier）：禁止特定类型的处理器重排序
